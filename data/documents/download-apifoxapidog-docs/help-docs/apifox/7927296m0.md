# 本地运行测试套件

## 选择合适的运行方式

Apifox 提供了多种测试套件运行方式，满足不同的需求：

1. **本地可视化运行**
   - 从本地机器发起，适用于小规模、快速的测试。
   - 适合同时开发和测试，便于实时查看结果和调整。

2. **本地 Apifox CLI 运行**
   - 适合大规模数据或迭代场景，运行速度较快。
   - 支持离线运行，适用于资源有限或不依赖图形界面的情况。

3. **CI/CD 运行**
   - 在 CI/CD 流水线中运行，适用于自动化集成和持续部署流程。
   - 适合频繁运行测试的环境，确保每次代码变更后都能及时验证 API 的稳定性。

4. **自托管 Runner 执行**
   - 团队可以在自己的服务器上部署 Apifox Runner，使用更强大的计算资源来运行测试。
   - 支持定时任务，适用于定期运行场景用例或负载较大的测试需求。


:::tip[]
测试套件中如果使用到了环境/全局变量，那么在选择不同方式运行时，实际使用的变量值可能会不同，导致运行结果不一致。[了解更多](https://docs.apifox.com/run-test-scenarios#%E4%B8%8D%E5%90%8C%E8%BF%90%E8%A1%8C%E6%96%B9%E5%BC%8F%E9%97%B4%E4%BD%BF%E7%94%A8%E4%B8%8D%E5%90%8C%E7%9A%84%E7%8E%AF%E5%A2%83%E5%85%A8%E5%B1%80%E5%8F%98%E9%87%8F%E5%80%BC%E7%9A%84%E8%A7%84%E5%88%99)
:::

### 通用配置

你可以在测试套件右侧的运行配置中调整以下设置：

- **运行环境**

  测试步骤中每个接口的运行链接（前置 URL），详情见 [环境与服务](https://docs.apifox.com/environments-and-services.md)。

- **运行模式**
  测试套件支持两种运行模式：
  - **串行**：按顺序逐个执行测试场景，前一个场景运行完成后再运行下一个。适用于场景之间存在依赖关系或资源受限的环境。
  - **并行**：多个测试场景同时执行，系统会根据机器的可用资源自动确定最佳并发数，无需手动调优。适用于大规模回归测试，可大幅缩短执行时间。
  
- **运行于**

  实际消耗硬件资源运行测试套件的机器。所有测试套件中发起的请求，会从这里指定的机器发起，所以有可能因为请求机器的网络环境不同导致产生不同的测试结果。
  
    :::caution[]
    此设置不会保存为测试套件的运行配置中，每次都是默认使用本机运行场景用例。并且这个设置不会在CLI 运行、定时运行时生效，使用以上方式运行时还是消耗当前机器资源发起请求。

    :::
  指定机器发起测试套件运行，如果该测试套件中使用了文件（请求发送文件、数据库连接、外部程序、SSL 证书等），则需要将所有要用到的文件存放在指定机器本地才可正常使用。

- **通知**

  是否在手动运行本测试套件完成之后，发送一个通知信息给指定的对象，信息内会有测试总览结果和详情链接。打开后可以设置是只要完成运行即进行通知还是在仅发生失败时进行通知，用以减少无效通知的打扰。通知对象的详细说明请 [参考此处](https://docs.apifox.com/notification-targets.md)。

## 并行运行

当测试套件中的场景用例数量增长到数百个时，串行执行会成为效率瓶颈。一个需要一小时才能完成的全量回归测试，会延迟发布流程，并降低生产监控中问题检测的响应速度。

将运行模式切换为 “并行” 后，系统会同时执行多个测试场景。并发数由系统根据当前机器的可用资源自动确定，无需手动调优。原本需要 60 分钟的回归测试可缩短至 30 分钟以内，而无需修改任何测试逻辑。

### 依赖隔离

并行执行时，每个测试场景都在独立的上下文中运行，确保：
- 一个场景中的共享变量不会影响其他场景
- 环境状态不会在场景之间产生干扰


:::tip[]
如果测试场景之间确实存在依赖关系（例如场景 B 需要使用场景 A 创建的数据），建议将它们合并到同一个测试场景中，通过顺序步骤来编排执行顺序。
:::

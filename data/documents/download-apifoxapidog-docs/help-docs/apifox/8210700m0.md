# 共用测试数据


在自动化测试中，许多场景用例可能需要使用相同的测试数据，例如用户登录信息、商品数据、配置参数等。Apifox 支持创建共用测试数据，可供多个场景用例共享使用，减少重复创建测试数据的工作，确保测试数据的一致性，让测试资源管理更加高效。

共用测试数据的主要优势：

1. **跨场景共享**

    在项目级别维护一份测试数据，多个场景用例可以直接引用，无需重复创建。

2. **统一管理**

    测试数据集中存放，修改一处即可同步到所有引用的场景用例，维护成本更低。

3. **一致性保障**

    确保不同场景用例使用相同的基础数据，避免因数据不一致导致的测试结果偏差。

4. **标准化测试流程**

    建立标准的测试数据规范，提升团队协作效率和测试流程的可维护性。

### 方式二：数据库连接测试数据


通过数据库连接方式创建的共用测试数据，可以直接从数据库中动态查询数据，适用于需要与真实数据保持一致或减少手动维护成本的场景。

<Steps>
  <Step>
在项目中点击左侧菜单栏中的 "自动化测试"，进入 "测试数据" 标签页，点击 "新建"，选择 **"新建测试数据（数据库连接）"**。
      
<Background>

![CleanShot 2026-02-26 at 14.18.14@2x.png](https://api.apifox.com/api/v1/projects/5097254/resources/625844/image-preview)
</Background>

  </Step>

  <Step>
若当前测试数据尚未配置数据库连接，会提示 “当前测试数据为空，请先配置数据库连接”，点击 "设置数据源配置"。
  </Step>

  <Step>
在弹窗中，选择已配置的数据库连接，或新建数据库连接。

在 SQL 命令中编写查询语句，用于获取测试数据，支持使用变量，例如：

```sql
SELECT id, name, email FROM pets;
````

<Background>

![02-Apifox.gif](https://api.apifox.com/api/v1/projects/5097254/resources/625852/image-preview)

  </Step>

  <Step>
确认配置无误后，点击 "保存"，完成基于数据库连接的共用测试数据创建。
  </Step>
</Steps>

:::tip[]
从数据库读取并存入测试数据后，测试数据就完全静态的了，不会跟着数据库内数据变化而变化。除非手动操作重新从数据库中获取数据。

## 管理共用测试数据

### 编辑共用测试数据

在 "共用测试数据" 列表中，点击数据名称即可进入编辑界面。你可以：

- 修改数据名称
- 添加、删除或修改数据行
- 新增或删除数据列（变量）
- 导入新数据覆盖现有数据
- 导出当前数据为 CSV 或 JSON 文件
:::tip[]
修改共用测试数据后，所有引用该数据的场景用例会自动使用最新的数据，无需手动同步。
:::


**数据库型共用测试数据的编辑限制说明：**

- 数据内容为**只读**，不支持直接编辑具体数据
- 支持手动刷新，从绑定的数据库中重新获取数据并覆盖
- 支持修改测试数据名称
- 支持更改数据库连接配置或 SQL 查询语句



### 按环境配置数据

共用测试数据支持按环境配置不同的数据集。你可以为开发环境、测试环境等分别维护一组数据，切换环境时会自动使用对应环境的数据值。

## 在脚本中使用共用测试数据

你可以在前置脚本或后置脚本中访问共用测试数据：

```js
// 检查共用测试数据变量是否存在
pm.iterationData.has("变量名")

// 获取共用测试数据变量的值
pm.iterationData.get("变量名")

// 将所有共用测试数据变量转为对象
pm.iterationData.toObject()
```

变量名需要与共用测试数据中的列名一致。

## 共用测试数据 vs 场景测试数据

| 对比项 | 共用测试数据 | 场景测试数据 |
|------|-------------|-------------|
| 作用范围 | 项目级别，可供多个测试场景使用 | 仅限当前测试场景使用 |
| 数据同步 | 修改后自动同步到所有引用场景 | 仅影响当前场景 |
| 适用场景 | 通用的基础数据，如用户信息、商品数据 | 特定场景专用的测试数据 |
| 维护成本 | 低，统一管理 | 高，需要在每个场景单独维护 |


## 最佳实践

**1. 合理划分数据粒度**

    将高复用性的基础数据设为共用测试数据，场景特定的数据保留在场景内。

**2. 使用清晰的命名规范**

    为共用测试数据使用描述性的名称，如 "用户登录数据"、"商品列表数据"，便于团队成员理解和选用。

**3. 定期清理无用数据**

    定期检查共用测试数据的引用情况，清理不再使用的数据，保持数据列表整洁。

**4. 配合环境变量使用**

    将环境相关的配置（如 URL、密钥）放在环境变量中，将业务数据放在共用测试数据中，实现关注点分离。


## 常见问题

<Accordion title="共用测试数据和环境变量有什么区别？" defaultOpen>
环境变量适合存储配置信息，如 API 地址、密钥等，通常每个变量只有一个值。共用测试数据适合存储业务数据，支持多行数据用于数据驱动测试。两者可以配合使用。
</Accordion>

<Accordion title="修改共用测试数据会影响正在运行的测试吗？">
不会。正在运行的测试会使用运行开始时的数据快照，修改只会影响后续新发起的测试运行。
</Accordion>

<Accordion title="共用测试数据支持多少条记录？">
共用测试数据支持存储大量记录，具体限制取决于你的团队套餐。建议单个数据集保持在合理范围内（如 1000 条以内），以获得最佳性能。
</Accordion>
